# Contributing to Laravel Overlord

Thank you for your interest in contributing to Laravel Overlord! We welcome contributions from the community.

## Code of Conduct

By participating in this project, you agree to maintain a respectful and inclusive environment for all contributors.

## How to Contribute

### Reporting Bugs

**Option 1: Built-in Bug/Issue Reporting Interface**

Laravel Overlord includes a built-in bug and issue reporting interface. You can use this interface to report bugs directly from your Laravel application.

**Option 2: GitHub Issues**

When the repository is published, you can report bugs via GitHub Issues. Please include:

- A clear description of the bug
- Steps to reproduce the issue
- Expected behavior
- Actual behavior
- Your environment (PHP version, Laravel version, etc.)
- Any relevant error messages or logs

### Suggesting Features

We welcome feature suggestions! Please use the built-in issue reporting interface or GitHub Issues (when available) to suggest new features. Include:

- A clear description of the feature
- Use cases and examples
- Potential implementation approach (if you have ideas)

### Pull Requests

When the repository is published, we'll accept pull requests. Please follow these guidelines:

1. **Fork the repository** and create a feature branch
2. **Follow coding standards** - use PSR-12 coding style
3. **Write tests** - ensure your changes are covered by tests
4. **Update documentation** - update README, SETUP.md, or other docs as needed
5. **Write clear commit messages** - follow conventional commit format
6. **Keep PRs focused** - one feature or bug fix per pull request

### Development Setup

1. Clone the repository (when published)
2. Install dependencies: `composer install`
3. Run tests: `phpunit`
4. Make your changes
5. Ensure all tests pass
6. Submit your pull request

### Coding Standards

- Follow PSR-12 coding style
- Use type hints where possible
- Add PHPDoc comments for public methods
- Keep functions focused and single-purpose
- Write self-documenting code with clear variable names

### Testing

- Write tests for new features
- Ensure existing tests still pass
- Aim for good test coverage
- Test edge cases and error conditions

### Documentation

- Update relevant documentation for new features
- Keep code comments clear and helpful
- Update CHANGELOG.md for user-facing changes
